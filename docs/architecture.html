Travel Agent MVP — Architecture

Streamlit → FastAPI → ADK Runner → root_agent → specialist agents/tools → Gemini. SQLite persists sessions on the side.

Streamlit UI (frontend/streamlit_app.py)
Pure HTTP client. Never imports google.adk or calls Gemini directly.
↓HTTP: session_id + message
FastAPI (backend/api/main.py)
Thin routes only — GET /health, POST /sessions, POST /chat. No agent logic here.
↓
agent_runner.py (backend/services)
One Runner + one DatabaseSessionService, created once per process, reused across every request.
↓
root_agent
Extracts requirements, validates duration, delegates, combines one final answer per the product contract.
tool: validate_trip_duration (deterministic, no LLM)
↓ calls as AgentTools
destination_agent
Recommends/validates destination → destination brief.
itinerary_agent
Day-by-day plan + costs.
tool: allocate_budget
↓
Gemini via Vertex AI
Every agent above is the same model, given a different role via its instruction text.
data/travel_agent.db (SQLite)
DatabaseSessionService reads/writes here. Sessions + events survive process restarts — this is what makes follow-up messages and app restarts work.

Key ideas worth remembering

  • AgentTool, not sub_agents. Specialists are called as tools, not handed the conversation — root_agent stays in control and always returns one combined answer, never a specialist's raw output.
  • Two unrelated "sessions." ADK's session (backend, SQLite, survives restarts) vs. Streamlit's st.session_state (frontend, one browser tab, resets on reload). The only link is the shared session_id string.
  • Tools are plain Python, agents are LLM calls. allocate_budget and validate_trip_duration run with zero model involvement — repeatable arithmetic, not guesses.
  • One Runner serves every session. session_id is passed per-call, not baked into the Runner, so a single backend process safely handles many independent conversations at once.
